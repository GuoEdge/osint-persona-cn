# 维护与改进方向（个人情报台）

本文档面向**后续维护者（人工或 AI）**，说明工具的核心价值、当前完成度、已知问题与分阶段改进路线。阅读顺序建议：本文 → `AGENTS.md` → `docs/CAPABILITIES.md`。

---

## 1. 工具定位

**OSINT Toolkit 是个人情报工作台，不是搜索器。**

| 维度 | 搜索器 | 个人情报台（本项目） |
|------|--------|---------------------|
| 目标 | 找到信息 → 结束 | 深度理解话题 → 沉淀知识 |
| 时间跨度 | 单次查询 | 多轮搜罗、跨时间追踪 |
| 用户参与 | 输入关键词 | 标注有用/噪音、构建画像、分叉深挖 |
| 积累价值 | 无 | 越用越懂用户、越用越准 |
| 产出 | 链接列表 | 审计产物 + AI 报告 + 研究树 + 知识库 |

**核心价值三要素**：

1. **越用越懂你** — 行为导入 → 心智画像 → 搜索时模拟"我会不会点这条" → 反馈修正
2. **越用越准** — 噪音标记影响下次搜罗、信源可信度积累、画像持续进化
3. **越用越有沉淀** — 知识库、研究树、跨轮综合、实体库

后续所有改进都应服务于这三个要素，**不加新信源、不加新页面，专注让已有的东西更可靠、更智能**。

## 3. 已知问题

以下问题从 git 历史分析和代码审查中发现，按影响排序：

### 3.1 反馈数据休眠（影响：高）

用户标了"噪音"/"有用"，数据写入 `~/.osint/feedback.jsonl`，但**不影响后续搜罗**。分叉深挖是唯一用到反馈的地方，且仅限单次会话。

- 噪音标记不降权相似 URL/域名
- 有用标记不提升同类条目相关度
- 无聚合分析（不知道哪个信源噪音率最高）

### 3.2 外部 API 无重试（影响：高）

B站/知乎/搜狗/DeepSeek 调用**失败即放弃或回退，无指数退避重试**。偶发 502/429 时用户看到"无结果"而非"暂时失败"。

### 3.3 配置无校验（影响：高）

4 层配置覆盖（`config.yaml` → `tunable_config` → `profiles` → `ai_directives`）**无 schema 校验**，写错 key 名静默忽略。用户改了设置不知道没生效。

### 3.4 进度状态机脆弱（影响：中）

进度状态分散在 `run_session.py`、`progress.py`、`timing_stats.py`、前端 SSE 四处，git 里有 3+ 个 commit 专门修进度卡住。

### 3.5 搜索质量不可度量（影响：中）

改了相关度规则/去重逻辑/过滤策略后，**没有指标判断是变好还是变坏**。只能靠人工抽检。

### 3.6 前端单文件（影响：中，维护成本）

`app.js` 7286 行无模块化，AI 改一处容易触发别处回归。

### 3.7 无数据迁移机制（影响：低，未来风险）

`knowledge.db` schema 升级靠 `CREATE TABLE IF NOT EXISTS` + 手动 `ALTER TABLE`，无版本号和迁移脚本。

## 4. 维护原则

### 4.1 测试是安全网

代码改动**必须跑对应层测试**（见 `AGENTS.md` 三层结构），不能只跑 `ruff check`：

- **第一层**（任何改动）：19 个文件，验证代码健全
- **第二层**（改采集/过滤）：15 个文件，验证数据质量
- **第三层**（改评论/行为）：5 个文件，验证评论抓取

### 4.2 提交粒度小

每个 commit 只改一件事，方便 `git revert`。如 `fix: B站 auth 短路` 与 `docs: 同步文档` 分开。

### 4.3 大改动分多次会话

不在一个 session 里改 collector + prompt + API + UI。一次改一个模块，跑完测试再继续。

### 4.4 文档同步

用户可见的行为变更（新约束、短路逻辑、配置项）必须同步更新 `docs/CAPABILITIES.md` 和 `docs/AI_CONTROL.md`。

### 4.5 AGENTS.md 是活文档

每次踩到新坑就在"AI 维护时的注意点"加一条，描述现象、根因、修复位置。

### 4.6 不加新功能

当前功能已经够用。后续精力投在**可靠性、闭环、可度量**上，而非新信源/新 AI 步骤/新页面。

## 5. 分阶段改进路线

### 第一阶段：堵漏 + 可靠性（75% → 82%）

目标：让已有的功能不再偶发失效。

| 项目 | 具体内容 | 涉及文件 |
|------|---------|---------|
| API 重试退避 | 所有外部调用加 3 次指数退避（1s/2s/4s） | `http/client.py`、各 collector |
| Cookie 过期前端提示 | auth 失败时前端显式 banner，不再静默空结果 | `bilibili.py`、`app.js` |
| 健康探测 | `GET /api/health/sources` 搜罗前快速探测各源可用性 | `web/routes/api.py` |
| 配置 schema 校验 | `tunable_config.py` 加 Pydantic 校验，非法值报错 | `services/tunable_config.py` |
| 设置回显 | Web 设置页保存后回显生效值 | `app.js` |
| 进度状态机统一 | 状态枚举 + 转换表集中管理，全路径测试覆盖 | `pipeline/progress.py`、`run_session.py` |
| `osint doctor` 配置检查 | 配置完整性、Cookie 有效性、DB schema 版本 | `cli.py` |

### 第二阶段：反馈闭环（82% → 86%）

目标：让用户反馈真正影响系统行为，工具越用越准。

| 项目 | 具体内容 | 涉及文件 |
|------|---------|---------|
| 噪音降权 | 标过噪音的 URL/域名/标题模式，下次搜罗自动降权 | `feedback/`、`analyzers/signals.py` |
| 有用加权 | 标过有用的条目，同类（同源/同关键词）相关度基础分提升 | `analyzers/signals.py` |
| 信源可信度 | 按用户历史反馈统计各信源噪音率，影响 source_plan 权重 | `ai/source_planner.py` |
| 反馈聚合分析 | `GET /api/feedback/summary` 返回各信源噪音率、高频噪音模式 | `feedback/store.py`、`web/routes/api.py` |
| 搜索质量指标 | 每次搜罗后记录有用/噪音比例到 run 产物 | `services/search.py` |

### 第三阶段：沉淀与跨轮（86% → 89%）

目标：让多次搜罗之间产生连接，知识有积累。

| 项目 | 具体内容 | 涉及文件 |
|------|---------|---------|
| 跨轮综合 | 搜罗时自动检查知识库/历史 run，提示"上月搜过相关话题" | `services/search.py`、`storage/` |
| 信息新鲜度追踪 | 同话题二次搜罗时，标注"自上次以来新增 N 条" | `services/search.py` |
| 画像持续进化 | 画像随行为变化自动微调，不只是 `auto_persona_rebuild` 全量重建 | `persona/builder.py` |
| 矛盾检测 | 多源说法矛盾时，报告结构化标注争议点 | `ai/report.py`、`ai/prompt_loader.py` |

### 第四阶段：深度洞察（89% → 92%）

目标：从"压缩信息"升级为"提炼洞察"。

| 项目 | 具体内容 | 涉及文件 |
|------|---------|---------|
| 实体提取 | 每次搜罗自动提取人名/产品/事件，形成实体库 | `ai/`、`~/.osint/entities/` |
| 趋势感知 | 同话题多次搜罗后，识别热度变化、新关键词 | `research/`、`ai/` |
| 观点图谱 | 不只"社区观点归纳"，而是"支持方 vs 反对方 + 论据" | `ai/`、`analyzers/comments.py` |

### 第五阶段：降低维护成本（持续）

| 项目 | 具体内容 |
|------|---------|
| app.js 模块化 | 按功能拆分（搜索/报告/研究树/知识库/设置），不改功能 |
| DB schema 版本 + 迁移 | 加版本号和迁移脚本，防未来改表结构丢数据 |
| DeepSeekClient 连接池 | AI 步骤复用连接，减少 7+ 次重复建连 |
| Collector 输出格式测试 | 验证 IntelItem 结构完整性，补测试空白 |

## 6. 优先级矩阵

```
高影响
  │
  │  ★ 反馈闭环              ★ API 重试
  │  (30%→65%)               (堵漏)
  │
  │  ★ 搜索质量指标          ★ 配置校验
  │  (可度量)                (防呆)
  │
  │  ○ 跨轮综合              ○ 进度状态机
  │  (沉淀)                  (可靠性)
  │
  │  ○ 实体提取              ○ app.js 拆分
  │  (洞察)                  (维护成本)
  │
  └─────────────────────────────────────── 低影响
   难                              易
```

**先做右上角（高影响 + 容易），再做左上角（高影响 + 难），最后做下方。**

## 7. 不做什么

明确列出**不做**的方向，避免后续维护者偏离：

- ❌ 不加新信源（7 个 + SERP 扩展已够）
- ❌ 不加新 AI 步骤（9 个已覆盖全链路）
- ❌ 不加新 Web 页面（9 个已覆盖所有功能）
- ❌ 不做多用户/账号体系（个人本地工具）
- ❌ 不做推荐流/主动推送（是情报工具不是推荐系统）
- ❌ 不做移动端（本地 Web UI 已够用）
- ❌ 不做实时监控（Watch 功能的 30 分钟间隔已够）

## 8. 衡量改进效果

每个阶段完成后，用以下指标检验：

| 指标 | 当前 | 目标 | 怎么测 |
|------|------|------|--------|
| 搜罗空结果率 | 未知 | <5% | 统计各源零结果频率 |
| 噪音标记率 | 未知 | 下降趋势 | 统计每次搜罗的噪音/有用比 |
| Cookie 过期感知时间 | 用户发现 | 即时 | 前端 banner 触发时间 |
| 设置不生效投诉 | 偶发 | 零 | 配置校验拦截率 |
| AI 摘要幻觉 | 偶发 | 零 | 用户纠错频率 |
